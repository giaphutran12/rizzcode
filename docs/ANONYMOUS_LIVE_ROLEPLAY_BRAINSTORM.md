# Anonymous Live Roleplay Brainstorm

Status: exploratory, not part of the approved RizzCode MVP.

Branch: `codex/anonymous-live-roleplay`

## Product idea

Let two adults practice one RizzCode scenario together in real time:

- The trainee plays himself.
- The practice partner plays a fictional counterpart from a private persona card.
- Both people know this is roleplay. Nobody is presented as an actual romantic match.
- The server owns the room transcript and the existing LLM judge scores the trainee after the conversation.

Working label: **Human Practice Room**.

## Ponytail MVP

Do not build public matchmaking yet. Start with an anonymous room code:

1. The trainee picks one of the ten existing scenarios.
2. RizzCode creates a short-lived room and shareable join code.
3. A second person joins as the practice partner.
4. The trainee sees the normal objective and observable context.
5. The partner sees a private fictional persona card, current mood, boundaries, and one simple goal.
6. They exchange three to six turns with typing, Sent, Delivered, and Seen states.
7. Either person can end immediately.
8. The existing server-side judge evaluates the canonical transcript.
9. The room expires and the human transcript is deleted. The trainee may save only the result locally.

This tests whether a real human counterpart is more useful than the AI persona without first owning a queue, ranking system, availability system, or moderation marketplace.

## Low-traffic strategy

- Default: invite code or share link.
- If no partner is available, keep the existing AI persona mode.
- Add a public queue only after concurrent usage proves that people regularly wait at the same time.

Future public matching can pair:

- one trainee seeking a practice partner;
- one volunteer willing to play a fictional counterpart;
- the same scenario and approximate difficulty;
- users who have not blocked each other.

## Safety contract

- Adults only.
- Fictional persona roleplay, not impersonation of a real woman.
- No photos, voice, file uploads, locations, social handles, phone numbers, or off-platform contact.
- Pseudonymous room names generated by the server.
- Visible Practice Partner label at all times.
- One-tap leave, report, and block.
- Server-side limits on message length, turn order, room lifetime, and participant count.
- Automatic rejection of personal contact details and severe abusive content.
- A boundary or participant exit ends the room immediately.
- No public transcript and no public score for the practice partner.

## Smallest technical shape

Keep the existing Express server and add no database for the first local comparison:

- `POST /api/live/rooms` creates a short-lived room.
- `POST /api/live/rooms/:code/join` claims the second seat.
- `GET /api/live/rooms/:code/events` streams room events with Server-Sent Events.
- `POST /api/live/rooms/:code/messages` accepts one server-validated turn.
- A bounded process-local `Map` owns room state, sequence, roles, expiry, and transcript.
- The judge reads only the server-owned room transcript.

Server-Sent Events plus normal POST requests are enough for two-person text practice. WebSockets, Supabase, accounts, queues, reconnection history, and distributed presence can wait until the room experiment earns them.

## Private partner card example

> You are Mai, playing a fictional person from the Keep the Thread Alive scenario. You are amused but busy. React to the trainee's actual message. You may warm up, stay neutral, lose interest, set a boundary, or leave. Do not coach him or reveal this card.

The card supplies direction, not exact dialogue. The practice partner writes every response.

## First acceptance test

The experiment is real when:

- two browser tabs join the same room with different roles;
- only the partner sees the private persona card;
- both sides see live typing and delivery states;
- messages cannot skip turns, cross rooms, or be rewritten;
- either side can end safely;
- judgment uses the exact server transcript;
- the room disappears after expiry;
- the existing AI practice mode still works unchanged.

## Not now

- Random public matchmaking
- Gender verification
- Dating profiles
- Photos, voice, video, gifts, GIF search, or file uploads
- Elo for practice partners
- Friend lists or direct messages
- Persistent human transcripts
- Payments
- Distributed room infrastructure

## Decision needed before implementation

Approve or reject this first slice:
